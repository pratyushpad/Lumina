# Lumina Model & Evaluation Card

Only measured numbers appear here; anything unmeasured is marked as such.
Current numbers: see [`docs/eval.md`](docs/eval.md) (regenerated by `make eval`).

## Models

| Role | Model | Serving | Notes |
|---|---|---|---|
| Embeddings | `all-MiniLM-L6-v2` | in-process (sentence-transformers) | 384-d, L2-normalized, cosine |
| Sparse retrieval | Okapi BM25 (`rank_bm25`) | in-process | shared lowercase alphanumeric tokenizer + stopword list on index & query sides |
| Sparse alternative | Postgres `ts_rank_cd` | in-SQL | not BM25; kept as an ablation lever and scale path |
| Reranker | `cross-encoder/ms-marco-MiniLM-L-6-v2` | in-process (CPU) | scores sigmoid-normalized for display & the refusal threshold |
| Generator (local) | `qwen2.5:7b-instruct-q4_K_M` | Ollama / vLLM via OpenAI-compatible API | target: RTX 5060 (~8 GB VRAM); tokens/sec measured per request — **no numbers reported until measured on that hardware** |
| Generator (hosted fallback) | `gemini-2.5-flash` | Google AI API | also serves vision (image description) |
| Eval judge | `gemini-2.5-flash` | Google AI API | see bias caveat below |

## Retrieval pipeline defaults

`hybrid_rrf` (BM25 + dense fused with RRF, k=60) over the session's documents,
top-50 candidates, cross-encoder rerank to top-5. Query transforms
(`multi_query`, `hyde`) exist but are **off by default** — multi-query measurably
added latency without moving any retrieval metric (see the ablation).

## Evaluation method

- **Dataset**: `eval/dataset.jsonl` — 42 answerable + 10 unanswerable questions,
  frozen. Answerable items were drafted by an LLM assistant *from the actual chunk
  contents* and each `relevant_chunk_ids` entry was verified to contain the answer
  before freezing. Binary relevance.
- **Corpus**: 4 documents (2 arXiv papers incl. figure-heavy PDF, 2 authored
  markdown docs), ingested deterministically (`make ingest`; sha256 doc ids,
  stable chunk ids, manifest committed procedure).
- **Retrieval metrics**: recall@5/10, MRR, NDCG@10 computed directly in
  `eval/metrics.py` (numpy); ablation runs call the production `RetrievalPipeline`
  in-process.
- **Generation metrics**: faithfulness, answer relevancy, context precision,
  context recall — the RAGAS metric *definitions* implemented as direct judge
  rubric prompts in `eval/judge.py` (no ragas library dependency; prompts are
  inspectable). Judge calls throttled for API rate limits; when a subset is used
  to respect daily quota, the `n` is reported in `docs/eval.md`.
- **Refusal calibration**: threshold sweep of the top rerank score over answerable
  + unanswerable questions; false-refusal and false-answer rates reported per
  threshold. `MIN_RERANK_SCORE` is chosen from that table.

## Known biases and limitations

1. **Self-evaluation bias**: the generator and the judge are both
   `gemini-2.5-flash`. Same-family judging systematically inflates scores. Treat
   absolute generation scores with skepticism; retrieval metrics and ablation
   *deltas* are the reliable signal. When answers are generated by the local Qwen
   model, Gemini judging becomes cross-model and this caveat weakens.
2. **Lexical bias in the eval set**: questions largely reuse source-chunk
   vocabulary, favoring BM25 over dense retrieval. Dense would close the gap on
   paraphrased queries; the eval set does not currently measure that.
3. **Small corpus**: 4 documents / 363 chunks. Latency numbers (p50 per stage) are
   not representative of large corpora; the BM25 index is in-memory and
   single-worker by design at this scale.
4. **Binary relevance**: NDCG uses binary gains; graded relevance judgments were
   not collected.
5. **tokens/sec**: local-model throughput is measured per request from streamed
   usage. Values marked with `~` in the UI are estimated from character counts
   when the backend returns no usage data.
